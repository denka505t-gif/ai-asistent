# 2026-07-21

## Сессия 1

Тема: архитектура субагентов для проекта «Сопоставление плана и факта строительных работ» (главная цель квартала, GOALS.md / PROJECTS.md №1)

### Сделано
- Денис спросил, какие специалисты (субагенты) нужны для еженедельного пайплайна: сбор плана по Объект/Участок/Вид работ/день из Excel от разных прорабов → приведение к формату → сбор факта → приведение к формату → сопоставление план/факт + причины отклонений → отчёт
- Предложил архитектуру из 3 субагентов:
  1. **Нормализатор** — приводит разнородный Excel (план ИЛИ факт — один навык, вызывается дважды) к единой таблице Объект/Участок/Вид работ/Дата/Значение. Самое узкое место всей задачи — форматы у разных прорабов могут сильно отличаться (порядок колонок, названия, объединённые ячейки, дни строками/колонками)
  2. **Аналитик отклонений** — джойнит нормализованные план+факт по ключу, считает отклонение, ранжирует, вытаскивает причины из файла или помечает «нужна причина от прораба»
  3. **Репортёр** — собирает финальный отчёт (Excel с подсветкой или текстовая сводка)
  - Шаги «сбор плана/факта с участков» — логистика, не субагент (прорабы присылают файлы, Денис кладёт в папку); опционально можно добавить диспетчера, который отслеживает кто не прислал

### Решения
- Проект живёт не в ai-asistent (это папка Агента), а в отдельной папке — заведём при переходе к реализации
- Следующий шаг — Денис присылает 2-3 реальных файла плана и 2-3 файла факта от разных участков, чтобы спроектировать точную схему нормализации, а не гадать на предположениях

- Уточнение: сбор плана/факта (шаги 1 и 3) через ТГ-бот. Ответ: папка на диске всё равно нужна — Нормализатор работает с файлами на диске, не с содержимым Telegram. Бот сам скачивает документ через Bot API и кладёт в структурированную папку (по участку/дате) — это обычный код бота, не ИИ-задача. Плюсы: бот знает участок по Telegram ID отправителя (не надо гадать по файлу), и может отслеживать кто не прислал → естественно закрывает роль «диспетчера» из прошлого пункта
- Важно на будущее: личный бот-Джарвис привязан к одному владельцу, для приёма файлов от прорабов (посторонние отправители) понадобится отдельный бот со своим токеном — не тот же самый

- Денис подтвердил: файл-коллектор (бот-приёмщик) добавляем в систему как четвёртую роль
- Уточнил для Дениса важную деталь: Приёмщик — не LLM-субагент как остальные три (не `.claude/agents/*.md`), а обычный код бота (Grammy: принял документ → скачал → сохранил в папку). Роль в пайплайне та же, но реализация другая — чтобы не закладывать неверные ожидания при сборке

### Решения
- Архитектура зафиксирована как 4 роли: Приёмщик (бот, код) → Нормализатор (субагент) → Аналитик отклонений (субагент) → Репортёр (субагент)

- Денис спросил про мессенджер eXpress (express.ms) для Приёмщика вместо Telegram. Проверил через веб-поиск: eXpress — корпоративный мессенджер, боты работают только на собственном сервере организации (CTS/eCTS), создать бота самостоятельно без доступа админа нельзя, аналога BotFather нет. Есть официальная Python-библиотека pybotx и API BotX — технически совместимо с нашим стеком (Python уже используем для Нормализатора)
- Уточнил у Дениса — своего сервера eXpress у компании нет

### Решения
- Приёмщика делаем на Telegram (как и планировали) — eXpress без готовой корпоративной инфраструктуры не оправдан ради одной функции сбора файлов
- eXpress отложен — рассмотреть, только если компания будет разворачивать его по другим причинам (не из-за этого проекта)

- Денис попросил альтернативы каналу сбора файлов без ВПН и без заморочек с СИБ, без человека вместо Приёмщика (Telegram в РФ сейчас работает нестабильно без ВПН). Проверил три варианта поиском:
  - **WhatsApp** — отпал: с декабря 2025 РКН системно ограничивает, на 15.01.2026 снятия ограничений не планируется — та же проблема с ВПН, что и Telegram
  - **VK** — самозаводится (сообщество + токен, без верификации), Bots LongPoll/Callback API, документы приходят как вложения, без ВПН и без СИБ
  - **MAX (max.ru)** — гос. мессенджер, официальный Bot API (dev.max.ru), но с августа 2025 публикация бота — только для верифицированных юрлиц РФ (для Дениса как директора компании решаемо, но лишний шаг); неясен охват прорабов, платформа новая
  - **Email (IMAP)** — без понятия «бот» вообще, скрипт по расписанию скачивает вложения, никакой регистрации нигде
