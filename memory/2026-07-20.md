## Сессия 2

Тема: VS Code Tunnel не подключался («агент не в сети», затем WebSocket 1006) — разобрали до конца, две реальные причины на сервере починили, третью (локальную) не стали

### Сделано
- Диагностика по систематическому дебагу (не гадали, шли по логам и evidence):
  1. «Агент не в сети» → на сервере служба `code-tunnel.service` (systemd `--user` под root, НЕ через шаблоны из `templates/vscode-tunnel/`) была жива 5 дней, но без единого сетевого соединения к релею Microsoft — тихо зависла. Перезапустил (`systemctl --user restart code-tunnel.service`) — подключение восстановилось
  2. WebSocket 1006 при повторной попытке → в логе `Remote - Tunnels.log` нашлась настоящая причина: диск сервера был забит на 99% (14 ГБ, свободно 164 МБ), VS Code не мог распаковать серверный компонент. Виновник — 6 старых версий VS Code Server в `/root/.vscode/cli/servers/` (~3,4 ГБ мусора, копятся при каждом обновлении VS Code на ноутбуке, сами не чистятся). Удалил старые версии (с подтверждения Дениса) — свободно 3+ ГБ
  3. WebSocket 1006 остался и после этого → по локальным логам ноутбука подтвердил, что это НЕ проблема установки VS Code (переустановка через winget уже пробовалась и не помогла) — общий признак у обоих сбоев: рвётся именно локальный WebSocket-мост на 127.0.0.1, а Remote-SSH (другой транспорт, тот же сервер) работает нормально. Заподозрил v2RayTun (TUN-режим, адаптер `sing-tun Tunnel` активен) — перехват loopback-трафика. Проверил и исключил системный прокси Windows (выключен) и `bypassLan` в конфиге v2RayTun (не про то). VPN трогать/отключать нельзя — без него не работает Claude Code
- Написал `knowledge/vscode-tunnel-troubleshooting.md` — обе серверные причины с командами восстановления + описание нерешённой локальной причины, чтобы не проходить путь заново в третий раз

### Решения
- Пункты 1 и 2 (сервер) — закрыты, зафиксированы в knowledge/
- Пункт 3 (локальный WebSocket 1006 из-за VPN) — **не чинить**, работать через Remote-SSH (Денис подтвердил). Дальше туннелем не пользоваться на этом ноутбуке
